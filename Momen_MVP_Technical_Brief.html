Momen is a mobile application that allows filmmakers to log timestamped markers during a shoot and export those markers in formats natively readable by professional editing software. The MVP is intentionally narrow in scope — it must do one thing perfectly before it does anything else.

Platform
iOS first. Swift or React Native are both acceptable. Must support iPhone running iOS 15 and above. iPad support is desirable but not required for MVP.
1

Session Management

A session represents a single shoot or shooting day

The user must be able to create a named session before logging begins. Each session stores all markers logged during that period.

  • Create a new session with a name and optional date
  • Complete a timecode sync step before marker logging is enabled (see section 3)
  • Open an existing session to review markers
  • Delete a session
Constraint
Sessions are stored locally on the device. Cloud sync is out of scope for MVP.
2

Marker Logging

The primary action of the entire application

When the user taps the main button, a marker is logged. Each marker must capture and store:

  • Adjusted timestamp — device clock time at exact moment of tap, offset-corrected, expressed in SMPTE format HH:MM:SS:FF
  • Marker number — sequential within the session (Marker 1, Marker 2…)
  • Optional note — short text attached to the marker, entered immediately after logging
  • Sync flag — boolean indicating whether the marker is a designated sync point
UX Requirements
  • Tap-to-mark action must be instantaneous — no perceptible delay
  • Button must be large, centred, and tappable with one thumb
  • Haptic feedback must fire on every successful log — UIImpactFeedbackGenerator at medium weight
  • Session view shows a chronological list: marker number, adjusted timecode, note
  • User can delete an individual marker from the list
3

Timecode Sync

Most critical component — marker logging is locked until complete

Critical
Without a reliable timecode reference, exported markers will not align to footage in the NLE and the product does not work. The sync step is mandatory.

On session creation, before the main logging screen is presented, the app must display a sync screen offering two methods.

Method A — Manual Offset Sync

User reads timecode from camera display and enters it. The moment they tap Sync, the app records:

  • Camera timecode as entered (T_camera)
  • Device clock at the instant of tap (T_device)
  • Offset = T_camera − T_device
Method B — Clap Sync

User taps marker button and claps in front of camera simultaneously. The resulting marker is stored as:

  • Label: SYNC
  • Flagged as sync point in session data
  • Editor completes offset alignment in NLE
Offset Formula
Marker timecode = T_tap + Offset

Every marker logged in a manual offset session must have the offset applied before the timecode is stored. The offset value must be stored against the session and displayed so the user can verify or re-trigger sync at any point. Re-syncing overwrites the previous offset for all subsequent markers; markers logged before the re-sync retain their original timecodes.

Sync Status Display
  • Which sync method was used (Manual Offset or Clap Sync)
  • For manual offset: the recorded offset value and time the sync was performed
  • For clap sync: a reminder that the editor must complete the alignment step
  • A Re-sync button allowing the user to repeat the sync at any point during the session
Clap Sync Export Note
The SYNC marker must appear as the first marker in all export files with the note: "Align this marker to the frame of the clap in your footage to synchronise all subsequent markers."
4

Timecode Display

Running counter visible on the main logging screen at all times

For manual offset sessions, the counter must display the offset-adjusted timecode in real time — the user should see the same timecode their camera is displaying, confirming sync is active. For clap sync sessions, the counter displays the raw device clock timecode.

Frame Rate Selection

User-selectable from settings. Must be stored against each session record — required for correct EDL generation at export.

23.976 24 25 29.97 30
Warning
An incorrect frame rate will cause markers to land on the wrong frame in the NLE. This is non-recoverable without a full re-export.
5

Export

Three formats generated simultaneously — all must work without manual reformatting

Format Target NLE Key Requirements
CSV Universal One row per marker. Columns: Marker Number, Timecode, Duration (default 1s), Note, Sync Point (TRUE/FALSE).
FCPXML 1.9 Final Cut Pro Must validate against Apple FCPXML 1.9 DTD. Each marker as a chapter marker. Note text as marker name. SYNC marker as first entry with editor instruction.
EDL (CMX 3600) Premiere Pro, DaVinci Resolve Single-frame edit event per marker. Note text in comment field. Drop-frame notation at 29.97fps. Non-drop-frame for all other rates. SYNC marker as first event.
Export Behaviour
  • All three formats generated simultaneously on export trigger
  • iOS share sheet presented — AirDrop to Mac, save to Files, or any share extension
  • Files named: SessionName_YYYYMMDD_markers.csv / .fcpxml / .edl
Clap Sync — Export Confirmation Screen
Must display before the share sheet: "Your export includes a SYNC marker. Share these files with your editor and ask them to align the SYNC marker to the frame of the clap in your footage. This will automatically align all subsequent markers to the correct position on the timeline."

Scope

In scope — MVP
  • Session management (local)
  • Timecode sync (manual + clap)
  • One-tap marker logging
  • Optional text notes per marker
  • Running timecode display
  • Frame rate selection
  • CSV export
  • FCPXML 1.9 export
  • EDL (CMX 3600) export
  • iOS share sheet
  • Haptic feedback
Out of scope — do not build
  • BLE hardware connectivity
  • LTC audio timecode decoding
  • QR code or slate scanning
  • Cloud sync or accounts
  • User authentication
  • AI transcription or voice notes
  • Role-specific dashboards
  • Team or multi-user features
  • GoPro API integration
  • In-app payment or subscription
  • Android support

Technical Constraints

  • Application must function entirely offline — no network connection required for any MVP feature
  • All data stored locally using Core Data or SQLite — no external database dependency
  • Offset calculation must use CACurrentMediaTime for highest precision — minimises drift between sync tap and first marker
  • FCPXML output must validate against the FCPXML 1.9 DTD before the build is considered complete — invalid FCPXML silently fails to import in Final Cut Pro
  • EDL must use drop-frame notation for 29.97fps and non-drop-frame for all other rates — incorrect notation causes timecode drift in the NLE
  • Application must be tested against all three NLEs with markers verified at correct timecode positions — both manual offset and clap sync sessions
  • UI should be functional and clean — design polish is phase two, functionality is the priority

Definition of Done

All 19 steps must pass without error before the build is considered complete

Manual Offset Sync Workflow — Steps 1–11
  1. 1User opens the app and creates a session named "Test Shoot"
  2. 2User is presented with the sync screen and enters a timecode of 01:00:00:00, tapping Sync immediately
  3. 3The session view displays the offset and confirms manual sync is active
  4. 4The running timecode display shows the offset-adjusted timecode in real time
  5. 5User logs 10 markers at irregular intervals, adding a text note to 3 of them
  6. 6User triggers export from the session view
  7. 7All three export files are generated and named correctly
  8. 8FCPXML imported into Final Cut Pro — all 10 markers at correct offset-adjusted timecode positions with notes visible
  9. 9EDL imported into DaVinci Resolve — all 10 markers at correct offset-adjusted timecode positions
  10. 10EDL imported into Premiere Pro — all 10 markers at correct offset-adjusted timecode positions
  11. 11CSV opened in Excel — all 10 markers present with correct timecodes and notes
Clap Sync Workflow — Steps 12–17
  1. 12User creates a second session and selects clap sync
  2. 13User taps the marker button — a SYNC marker is created as the first marker in the session
  3. 14User logs 5 further markers
  4. 15User exports — all three files generated with SYNC marker as first entry and editor instruction in note field
  5. 16Export confirmation screen displays the clap sync editor instruction before the share sheet appears
  6. 17Editor import instruction verified to be clear and actionable by a person who has not used Momen before
Environment — Steps 18–19
  1. 18The entire manual offset flow repeated on a device that has never had the app installed, with no internet connection
  2. 19The entire clap sync flow repeated on a device that has never had the app installed, with no internet connection

Handoff Notes

  • The sync implementation and export validation against real NLEs are the two highest risk items in this build. Treat them as such from day one.
  • The offset calculation must be tested over extended sessions to confirm there is no drift. A one-frame drift over a ten-hour shoot is acceptable. A one-second drift is not.
  • The FCPXML specification has edge cases that are not well documented and will require careful testing. Budget additional time for NLE validation specifically — this is where most implementations of this type encounter unexpected failures.
  • The UI requires no design assets for MVP. A functional wireframe is sufficient. Brand and visual identity are applied in version two.
